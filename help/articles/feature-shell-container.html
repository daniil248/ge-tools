Концепция

В реальном проекте часто несколько одинаковых стоек: SR01, SR02, ..., SR08. Создавать 8 отдельных карточек на схеме — захламляет canvas. Решение: группа потребителей-контейнер, где одна карточка представляет N экземпляров.

Модель «Shell»

Один из реальных потребителей (с самым младшим обозначением — например SR01) становится оболочкой (shell). Остальные (SR02..SR08) числятся «внутри». На схеме видна только оболочка — карточка с маркером ×N (стопка карточек с peek).

Свойства наследуются

По модели пользователя: «свойства расположенных внутри объектов связаны с основным (кроме обозначений)».

  • Tag/name — индивидуальные для каждого экземпляра
  • demandKw, cosφ, фаза, напряжение, subtype — наследуются от shell
  • В инспекторе экземпляра электрические поля показаны read-only с маркером «🔗 Электрические параметры наследуются от оболочки»

Изменения в shell автоматически распространяются на все экземпляры — через getter-helpers (effectiveDemandKw, effectiveCosPhi и т.д. в shared/zones.js).

Как создать связь

Drag-drop на canvas

Перетащите один consumer-узел поверх другого — если параметры совместимы, появляется alias-link. Source-узел исчезает с canvas (становится «внутри» shell), shell получает счётчик ×2 (затем ×3, ...).

Через Группа-tab инспектора

Откройте свойства любого consumer-узла (групповой потребитель) → вкладка 👥 Группа:

  • Список «слотов» — позиций внутри shell. Каждый слот может быть привязан к конкретному экземпляру или быть anonymous.
  • Picker «🔗 Связать с существующими» — список одиночных consumer'ов, которые можно перетащить в свободный слот.
  • Кнопки на linked-слоте: ↗ Извлечь (split-out — слот удаляется, count--), ✂ Разъединить (slot становится anonymous).

Click-навигация

В Группа-tab каждая строка слота имеет:

  • Click по строке (вне кнопок) → открыть свойства привязанного узла
  • Click по 🔗 (зелёный кружок) → центрировать камеру на shell на схеме (если узел на другой странице — переключение)
  • Кнопки ↗ (split-out) / ✂ (unlink)

Также: Ctrl+click по объекту в Реестре → центрирование на схеме (см. навигацию в Конструкторе).

Auto-tag = младшее обозначение

Эффективное обозначение группы (отображаемое на канвасе и в модалке) = обозначение первого экземпляра в естественной сортировке (SR01 < SR02 < SR10 через localeCompare numeric:true). При добавлении SR00 — оно автоматически становится новым «лицом» группы.

⚠ Raw tag группы (внутреннее n.tag) не меняется — остаётся уникальным для save/load. Меняется только display-derived через effectiveTag().
