Файловое хранилище проектов

Как drawio: проект сохраняется не в браузере или облаке, а в локальном файле .raschet.json. Файл можно положить на сетевой ресурс и совместно работать с коллегами (один редактирует, остальные видят).

Зачем это нужно

  • Без квот Firebase. Бесплатный (Spark) план Firestore имеет суточный лимит на запись. При исчерпании получаете ошибку «Quota exceeded» и сохранение перестаёт работать. Файловый режим работает без ограничений.
  • Совместное использование без сервера. Положили файл на общий сетевой диск (Z:\projects\... или \\server\share\...) — коллеги открывают его прямо оттуда.
  • Полная privacy. Данные не покидают вашу машину / корпоративную сеть. Никаких облачных серверов.
  • Резервное копирование тривиально. Просто скопировать файл.
  • Версионный контроль (git, SVN). Текстовый JSON легко diff-ится и мерджится.

Где найти кнопки

Sidebar (раздел «Файл») — три кнопки:

  • 📁 Открыть файл проекта… — выбрать существующий .raschet.json с правом записи. После открытия в badge sidebar'а появится «📁 имя файла (in-place save)».
  • 👁 Открыть только для чтения… — открыть файл без права записи. Полезно для коллег-наблюдателей на общем ресурсе.
  • 💾 Сохранить в файл… — сохранить текущую схему в новый файл (через диалог «Save As»). После сохранения дальнейшие правки автоматически пишутся в этот файл.

Сценарий 1. Одиночная работа

  1. Откройте Конструктор. Создайте схему как обычно.
  2. Sidebar → «💾 Сохранить в файл…». Выберите место (например, D:\projects\my-dc.raschet.json).
  3. Дальше работайте как всегда — каждые 1.5 секунды после изменения файл автоматически перезаписывается. Кнопка «Сохранить» в шапке тоже работает (вручную).
  4. В badge sidebar'а видно: 📁 my-dc.raschet.json (in-place save).
  5. В следующий раз: «📁 Открыть файл проекта…» → выбираете тот же файл → продолжаете работать.

Сценарий 2. Командная работа на сетевом ресурсе

Подготовка:

  • Положите файл проекта на общий сетевой диск, который виден у всех. Например Z:\projects\my-dc.raschet.json (mapped drive Windows) или \\fileserver\projects\my-dc.raschet.json.
  • Договоритесь, кто сегодня «писатель» — только один человек одновременно.

Писатель (тот, кто редактирует):

  1. Sidebar → «📁 Открыть файл проекта…» → выбирает Z:\projects\my-dc.raschet.json.
  2. Браузер запросит разрешение на запись — нажать «Allow».
  3. Работает как обычно. Каждое изменение через 1.5 с пишется в файл на сетевом ресурсе.

Читатели (наблюдатели, ГИП на ревью, заказчик):

  1. Sidebar → «👁 Открыть только для чтения…» → выбирают тот же файл.
  2. Видят актуальное содержимое. Могут локально что-то менять для проверки сценариев — изменения не запишутся в файл, останутся только в браузере.
  3. Чтобы увидеть свежие правки писателя — закройте проект и откройте файл заново (или нажмите F5 в браузере).

In-place save работает через File System Access API, который в Firefox/Safari пока не реализован (на 2026 год). В этих браузерах файл будет каждый раз скачиваться заново, а открытие — через стандартный диалог выбора файла.

Формат файла

{
  "_format": "raschet-project",
  "_version": "1",
  "_savedAt": "2026-05-06T12:34:56.789Z",
  "name": "Project name",
  "customer": "ABC Corp",
  "address": "Москва, ул. Примерная, 1",
  "code": "PR-2026-001",
  "info": "...",
  "scheme": {
    "nodes": [...],
    "conns": [...],
    "zones": [...],
    "pages": [...],
    "modes": [...]
  }
}

Поддерживается также legacy-формат — обычный JSON-экспорт (только scheme в корне без _format) — он автоматически распознаётся при открытии.

Конфликты при совместной работе

В текущей версии нет file-locking — если двое одновременно открыли один файл с правом записи, последний save перезатрёт изменения первого. Поэтому правило: один писатель, остальные — read-only.

Для надёжного совместного редактирования (одновременная работа нескольких авторов) — используйте облачный режим с Firebase, где есть merge-conflict-detection.

  • Хранение схем (storage keys) — где какие данные лежат
