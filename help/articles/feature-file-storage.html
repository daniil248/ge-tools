Файловое хранилище проектов

Как drawio: проект сохраняется не в браузере или облаке, а в локальном файле .raschet.json. Файл можно положить на сетевой ресурс и совместно работать с коллегами (один редактирует, остальные видят).

Зачем это нужно

  • Без квот Firebase. Бесплатный (Spark) план Firestore имеет суточный лимит на запись. При исчерпании получаете ошибку «Quota exceeded» и сохранение перестаёт работать. Файловый режим работает без ограничений.
  • Совместное использование без сервера. Положили файл на общий сетевой диск (Z:\projects\... или \\server\share\...) — коллеги открывают его прямо оттуда.
  • Полная privacy. Данные не покидают вашу машину / корпоративную сеть. Никаких облачных серверов.
  • Резервное копирование тривиально. Просто скопировать файл.
  • Версионный контроль (git, SVN). Текстовый JSON легко diff-ится и мерджится.

Где найти кнопки

Через страницу /projects/

На странице списка проектов в верхней панели зелёная кнопка «📁 Открыть файл (drawio)» — самый быстрый вход в file-mode без необходимости предварительно открывать какой-либо проект.

В Конструкторе (sidebar → раздел «📁 Локальный файл»)

  • 📁 Открыть файл проекта… (хоткей Ctrl+O) — выбрать существующий .raschet.json с правом записи. После открытия в badge sidebar'а появится «📁 имя файла (auto-save) · HH:MM».
  • 👁 Открыть только для чтения… — открыть файл без права записи. Полезно для коллег-наблюдателей на общем ресурсе.
  • 💾 Сохранить в файл… (хоткей Ctrl+Shift+S) — сохранить текущую схему в новый файл (через диалог «Save As»). После сохранения дальнейшие правки автоматически пишутся в этот файл.

Кнопки внутри file-mode badge

Когда file-mode активен, в самом badge'е доступны:

  • ↻ Перечитать — перечитать содержимое файла с диска. Полезно когда коллега в режиме «писатель» сохранил свежие правки на сетевом ресурсе. Ваши локальные правки потеряются.
  • ✕ Закрыть файл — выйти из file-mode. Схема в браузере остаётся, но автосохранение в файл прекращается.

Persistent file handle (запоминание файла после reload)

После первого открытия файла handle сохраняется в IndexedDB браузера. Если вы перезагрузите страницу или вернётесь в приложение завтра, в badge sidebar'а появится подсказка «📁 Последний файл: имя · N мин назад» с двумя кнопками:

  • ↻ Открыть снова — браузер запросит разрешение на запись (один клик), и file-mode восстановится.
  • ✕ — забыть этот файл, badge скроется.

Без этой возможности нужно было бы каждый раз заново выбирать файл через showOpenFilePicker.

Обнаружение внешних изменений

Каждые 30 секунд (а также сразу при возврате на вкладку через visibility-change) приложение проверяет mtime файла на диске. Если кто-то записал в файл извне (например, коллега-«писатель» на сетевом ресурсе):

  • Появляется toast «⚠ Файл изменён извне».
  • Badge подсвечивается красным с пометкой «⚠ изменён извне».
  • Для read-only — мягкая подсказка «нажмите ↻ Перечитать».
  • Для writer — жёсткое предупреждение «следующее автосохранение перезапишет внешние правки!».

Эта проверка локальная (handle.getFile() — обычный read с диска) и не нагружает Firestore-квоту.

Сценарий 1. Одиночная работа

  1. Откройте Конструктор. Создайте схему как обычно.
  2. Sidebar → «💾 Сохранить в файл…» (или Ctrl+Shift+S). Выберите место (например, D:\projects\my-dc.raschet.json).
  3. Дальше работайте как всегда — каждые 3 секунды после изменения файл автоматически перезаписывается. Кнопка «Сохранить» в шапке тоже работает (вручную).
  4. В badge sidebar'а видно: 📁 my-dc.raschet.json (auto-save) · 14:32.
  5. В следующий раз: «📁 Открыть файл проекта…» (или Ctrl+O), либо нажмите «↻ Открыть снова» в auto-restore badge'е.

Сценарий 2. Командная работа на сетевом ресурсе

Подготовка:

  • Положите файл проекта на общий сетевой диск, который виден у всех. Например Z:\projects\my-dc.raschet.json (mapped drive Windows) или \\fileserver\projects\my-dc.raschet.json.
  • Договоритесь, кто сегодня «писатель» — только один человек одновременно.

Писатель (тот, кто редактирует):

  1. Sidebar → «📁 Открыть файл проекта…» → выбирает Z:\projects\my-dc.raschet.json.
  2. Браузер запросит разрешение на запись — нажать «Allow».
  3. Работает как обычно. Каждое изменение через 3 секунды пишется в файл на сетевом ресурсе.

Читатели (наблюдатели, ГИП на ревью, заказчик):

  1. Sidebar → «👁 Открыть только для чтения…» → выбирают тот же файл.
  2. Видят актуальное содержимое. Могут локально что-то менять для проверки сценариев — изменения не запишутся в файл, останутся только в браузере.
  3. Когда писатель сохранит свежие правки → читатели увидят toast «⚠ Файл изменён извне» (мгновенно при возврате на вкладку, или максимум через 30 секунд). Нажмут «↻ Перечитать» в badge'е → подгрузят актуальное содержимое.

In-place save и persistent handle работают через File System Access API, который в Firefox/Safari пока не реализован (на 2026 год). В этих браузерах файл будет каждый раз скачиваться заново при сохранении, а открытие — через стандартный диалог выбора файла без запоминания.

File-mode и Cloud-mode — взаимоисключающие

Чтобы избежать data corruption, file-mode и cloud-режим не могут быть активны одновременно:

  • Если вы открываете cloud-проект пока был активен file-mode → file-mode автоматически выключается, handle забывается.
  • Если вы открываете файл пока работали с cloud-проектом → collab останавливается, шапка очищается, state сбрасывается, начинается чистая file-mode сессия.

Поддерживается также legacy-формат — обычный JSON-экспорт (только scheme в корне без _format) — он автоматически распознаётся при открытии.

Конфликты при совместной работе

В текущей версии нет file-locking — если двое одновременно открыли один файл с правом записи, последний save перезатрёт изменения первого. Поэтому правило: один писатель, остальные — read-only. External-change detection хотя бы оповещает об этом — если вы вторым открыли с правом записи, то в течение 30 секунд после первого save коллеги получите красный warning.

Для надёжного совместного редактирования (одновременная работа нескольких авторов) — используйте облачный режим с Firebase, где есть merge-conflict-detection и presence.
